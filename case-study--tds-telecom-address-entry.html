TDS Telecom Address Entry Experience Case Study

The Challenge

TDS Telecom serves almost 900 rural, suburban, and metropolitan communities in over 25 states across the U.S. For the past several years, TDS was worked to increase its reach by acquiring cable companies, building networks in new territories, and expanding existing build areas. While the customer facing websites and marketing materials have been visually updated to match the TDS brand, potential TDS customers must order their services at a specific website depending upon their address.

Unfortunately, our site visitors were having trouble getting to the right website for their experience. The backend systems also took too much time looking up the entered address so that the site can display the correct messaging in order to better direct the user on where to go.

The Work

Write User Flows

Before the software developers could finish their work on the backend, I had to document all of the user types and scenarios the system was expected to handle. After several meetings and a lot of collaboration with IT, the project owner, and marketing, the final documentation captured over 100 user flows. Because of the number of flows and the project timeline, flow diagrams were only used for some of the more complicated cases.

Design Intercept Pages

After the user flows were written, we had to decide how the system would let users know that they had to finish their experience on another site. We kept the messaging short and I used yellow alerts as a visual cue whenever customers were being directed to leave their current site. We also had to make sure that the proper analytics were in place so that we would know whether our work was successful. To help with that, each intercept had its own unique URL to make tracking easier in Google Analytics.

Document Component Requirents

All of the main websites are built using the Adobe Experience Manager (AEM) content management system (CMS). In order to make sure we could quickly make content or copy changes to the intercept pages if we find that users are struggling with the experience, we have to put that content into AEM components. While a lot of the components were created before this project, I had to work closely with the software developer to define and document what the component will allow the site author to edit. I also have to supply HTML and CSS at this stage by directly contributing to the Github code repository.

Quality Assurance Testing

We didn't have a dedicated QA team to help out with testing on this project so I had to make sure I was extra diligent about checking my own work before handing things off to IT as well as checking the work again after IT implementation in the development environment.

The Results

Our web analyst compared data 30 days prior to project launch and 30 days after project launch. The results were extremely positive with the largest impact in traffic flowing from tdsfiber.com to tdstelecom.com.

Referral visits increased by 31% and orders increased by 42%.

Other positive results include increased traffic conversions to tdstelecom.com from other sites.

Referral visits from hellotds.com increased by 71%.

The team worked extremely well together and met the deadline by staying in constant communication throughout the life cycle of the project. Because of the improvement to the address entry experience, users were given faster feedback and more clarity on which site would meet their needs. We originally predicted the conversion rates (percentage of visits that turn into web orders) would improve by 2% but we exceeded that goal several times over.
